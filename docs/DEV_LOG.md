## Этап 6. Полноценный плеер: пауза, стоп, скорость, подсветка, перемотка
- `js/tts.js` существенно расширен. Введено внутреннее состояние `ttsState`
  (isPlaying, isPaused, fullText, currentCharIndex, rate, wordSpans,
  wordOffsets, container, utteranceStartOffset, onStateChange).
- Новая функция `renderText(text, container)`: раскладывает текст пересказа
  на токены через регулярку `/(\S+|\s+)/g`, оборачивает слова в
  `<span class="word" data-offset data-index>`, пробельные куски кладёт
  как текстовые узлы (чтобы абзацы и пунктуация сохранились). Заполняет
  `wordSpans` и `wordOffsets` для подсветки и перемотки.
- Подсветка: `highlightWordAt(charIndex)` ищет слово, чьё смещение
  ближе всего к текущей позиции, добавляет класс `.word--active`, мягко
  скроллит к нему через `scrollIntoView`, если оно ушло за пределы вьюпорта.
  Работает через событие `utterance.onboundary` — браузер сам присылает
  `charIndex` читаемого слова.
- Управление воспроизведением: `pause()`, `resume()`, `stop()`, `setRate()`.
  При смене скорости во время чтения — перезапуск с текущей позиции.
- Перемотка `rewind(seconds)`: считает ≈2.5 слова/сек × текущая скорость,
  отступает на нужное число слов назад от `currentCharIndex` и запускает
  заново через `startUtteranceFrom(offset)`. Точное смещение знаем благодаря
  `onboundary`, поэтому перемотка попадает в слово, а не в случайное место.
- Старт с произвольной позиции: `speakFromWordIndex(wordIndex)` — для тапа
  по любому слову в тексте.
- `onTtsStateChange(callback)`: подписка из `app.js`, чтобы UI
  переключал кнопки между «▶ Слушать / ⏸ Пауза / ▶ Продолжить» и
  показывал/скрывал «⏹ Стоп» и «⏪ −15 сек» по факту состояния плеера.
- Защита от отсутствия русского голоса: если `pickRussianVoice()` вернул
  `null`, а голоса в системе всё-таки есть (значит, русского просто нет) —
  показываем alert и не запускаем озвучку. Английским читать русский
  текст бессмысленно: голос спотыкается на первом же слове с цифрой
  (поймали это на Яндекс.Браузере, где русского голоса в системе нет).
- Защита от гонки при первом запуске: если `getVoices()` вернул пустой
  массив, ждём событие `voiceschanged` и перезапускаем `startUtteranceFrom`.
- `index.html`: в `.result__controls` добавлены кнопки `#stop-btn`,
  `#rewind-btn` (оба `hidden` по умолчанию) и `<select id="rate-select">`
  со значениями 0.75 / 1 / 1.25 / 1.5 / 2.
- `js/app.js`: главная кнопка `#speak-btn` стала трёхрежимной (Слушать ↔
  Пауза ↔ Продолжить) — состояние определяется через `getTtsState()`.
  Текст пересказа теперь отдаётся в `tts.js` через `renderText`, а не
  пишется в `textContent` напрямую. Делегированный обработчик клика на
  `#result-text`: если кликнули по `.word` — запускаем
  `speakFromWordIndex(parseInt(target.dataset.index))`. На клик «Получить
  пересказ» дополнительно вызываем `stop()`, чтобы остановить прошлое
  чтение.
- `css/styles.css`: `.result__text .word` — кликабельные слова с лёгким
  hover-фоном; `.word--active` — мягкий жёлтый `#fde68a` (как в Audible
  / Speechify); `.result__controls` стал flex-wrap, `.rate-control`
  прижат к правому краю через `margin-left: auto`. `button.secondary`
  получил `white-space: nowrap`, чтобы кнопки не ломались на узких экранах.
  `line-height` основного текста увеличен с 1.55 до 1.7, чтобы подсветка
  не «склеивала» строки.
- Совместимость: десктопный Яндекс.Браузер без русского голоса —
  корректно показывает alert. На устройствах с русским голосом

  после деплоя на GitHub Pages.
- Решение по UX: одна кнопка перемотки «−15 сек» (а не две). Крупные
  прыжки удобнее делать тапом по слову — это надёжнее и точнее.

### Этап 6. Доработки после полевых испытаний на Android Chrome
- `speechSynthesis.pause()` / `resume()` на Android Chrome не работают —
  pause фактически делает cancel, resume молчит. Эмулировали: при паузе
  фиксируем `currentCharIndex` и делаем cancel; при «продолжить» —
  заново `startUtteranceFrom(currentCharIndex)`. Для пользователя выглядит
  как обычная пауза.
- Событие `utterance.onboundary` на Android Chrome тоже не приходит
  (известный баг Chromium issue 521666). Из-за этого без фолбэка
  не работала ни подсветка, ни перемотка (currentCharIndex оставался 0,
  «−15 сек» отматывал в начало текста).
- Решение: в `tts.js` появилась эмуляция подсветки по таймеру. Через 500 мс
  после старта реплики проверяем флаг `boundaryReceived`. Если ни одного
  boundary не пришло — запускаем `setInterval(200)`, который вычисляет
  позицию по формуле `(Date.now() - utteranceStartTime) / 1000 *
  CHARS_PER_SECOND * rate`. Константа CHARS_PER_SECOND = 14 (подобрано
  под русскую речь). Если boundary всё-таки придёт — таймер гасится,
  переключаемся на точную подсветку. Точность эмуляции — ±2–3 слова, для
  «не потерять нить» этого достаточно.
- Аналогичная оценка currentCharIndex по времени применяется в `pause()`
  и `rewind()`, если ни boundary, ни таймер не успели обновить позицию.
- `onstart` на Android Chrome тоже ненадёжен — состояние `isPlaying = true`
  ставим синхронно при вызове `startUtteranceFrom`, не дожидаясь
  подтверждения от браузера. Иначе UI не переключал кнопки.
- Порядок настройки `SpeechSynthesisUtterance` важен: сначала voice + lang,
  потом rate. Если rate выставить ДО voice — на некоторых сборках Android
  Chrome он игнорируется и читает на скорости 1.0.
- Вёрстка плеера переделана дважды. Финальное решение: `.result__controls`
  — relative-контейнер с `padding-right: 80px`, `.rate-control` —
  absolute в правом верхнем углу. Кнопки слева текут как обычный flex,
  селект всегда на месте, ничего не прыгает при появлении/исчезновении
  Пауза/Стоп/−15 сек. Подпись «Скорость» убрана — селект «1× ▾» сам по
  себе читается однозначно, для скринридеров оставлен `aria-label` на
  `<label>`.
- Финальная проверка на Android Chrome (Xiaomi): чтение, подсветка
  (эмулированная, бежит с лёгким опережением/отставанием — норма),
  пауза/продолжить, стоп, перемотка −15 сек, смена скорости 0.75–2×,
  тап по любому слову для старта с него — всё работает.

## Этап 8. Модуль api.js: запрос к OpenRouter и обработка ошибок
- Создан `js/api.js`. Главная функция `requestSummary(systemPrompt, userQuery)`:
  проверяет наличие ключа (`hasApiKey`) и выбранной модели (`getModelId`),
  шлёт POST на `https://openrouter.ai/api/v1/chat/completions`. Заголовки:
  `Authorization: Bearer <ключ>`, `Content-Type`, плюс `HTTP-Referer` и
  `X-Title` (OpenRouter просит указывать источник). Тело: `{model, messages}`,
  где messages — системный промпт + реплика пользователя.
- Модуль ничего не рисует на экране: возвращает строку с ответом модели
  либо бросает Error с кодом-меткой. UI-связка будет на этапе 10.
- Обработка ошибок по HTTP-кодам: 401 → BAD_KEY, 402 → MODEL_PAID,
  403 → MODEL_BLOCKED, 404 → MODEL_NOT_FOUND, 429 → RATE_LIMIT,
  прочее → MODEL_ERROR; сетевой сбой → NETWORK; пустой ответ → EMPTY_RESPONSE.
  Функция `describeApiError(code)` переводит метку в человеческий текст.
- Принцип ТЗ соблюдён: при отвале бесплатной модели НИКОГДА не подменяем
  на платную — честно говорим «выбери другую модель в настройках».
- `index.html`: `js/api.js` подключён между `storage.js` и `app.js`
  (порядок важен — api.js использует функции storage.js).
- Тест через консоль браузера на проде (без подключения к кнопке):
  `requestSummary(...).then/.catch`. Результаты:
  - Модели Google (Gemma 4 31B, Gemma 4 26B) → 403 «Blocked by Google AI
    Studio»: заблокированы по региону (РФ). Штатный кейс — обходится VPN
    либо выбором другой модели.
  - Nemotron 3 Super 120B и MiniMax M2.5 → отвечают корректно
    (вернули осмысленный текст про Пушкина).
  - Связь с OpenRouter подтверждена.
- Решение: список `models.json` не меняем. При недоступности Gemma по
  региону пользователь либо включает VPN, либо выбирает Nemotron/MiniMax.


## Этап 9. Модуль prompts.js: системный промпт-«литературовед»
- Создан `js/prompts.js` — глобальная константа `LITERATURE_SYSTEM_PROMPT`.

  Хранится как template literal (обратные кавычки) — длинный текст с
  переносами строк, абзацы сохраняются.
- Что задаёт промпт: роль (преподаватель русской литературы готовит
  абитуриента ВГИКа к устному экзамену), правило объёма (потолок 1500 слов

  150–350 слов на короткое стихотворение до 1000–1500 на роман/пьесу),
  адаптивную структуру ответа из 6 пунктов, формат без Markdown (текст
  идёт прямо в синтезатор речи — никаких #, *, -, списков, заголовков),

  несуществующее название), модель должна вернуть ровно строку

- `index.html`: `js/prompts.js` подключён между `api.js` и `app.js`.
  Сам по себе ни от кого не зависит, нужен только для `app.js`.

## Этап 10. Реальная генерация: ввод → LLM → пересказ → озвучка
- Из `js/app.js` удалена заглушка `FAKE_SUMMARY`. Кнопка «Получить пересказ»
  стала асинхронной (`async`): блокируется и показывает «Готовлю пересказ…»,
  вызывает `requestSummary(LITERATURE_SYSTEM_PROMPT, query)` из api.js,
  обёрнута в `try/catch/finally` (в finally кнопка возвращается в строй).

## Этап 12. Service Worker: офлайн-доступ и версионирование кэша
- Создан `service-worker.js` в КОРНЕ проекта (рядом с index.html, не в js/).
  Кэширует оболочку приложения для офлайн-старта.
- Версионирование кэша: `CACHE_VERSION = 1`, имя кэша `deliu-cache-v1`.
  ГЛАВНОЕ ПРАВИЛО на будущее: при любой правке CSS/JS поднимать
  CACHE_VERSION на 1 перед push — старый кэш удаляется в `activate`,
  телефон скачивает всё заново. Заменяет ручной `?v=N`.
- Список оболочки (APP_SHELL): index.html, manifest.json, css/styles.css,
  все 5 js-файлов (tts, storage, api, prompts, app), data/models.json,
  обе иконки (192/512). Пути с базой `/deliureader/`. addAll — если хоть
  один файл не скачался, установка падает целиком (поэтому в списке только
  реально существующие файлы).
- Стратегии кэширования:
  - HTML/CSS/JS/JSON — network-first (сначала сеть, кэш запасной): пока
    интернет есть, всегда свежий код; офлайн — из кэша.
  - Картинки (request.destination === 'image') — cache-first: меняются
    редко, нет смысла каждый раз ходить в сеть.
  - POST не трогаем (запросы к OpenRouter всегда идут в сеть напрямую).
  - Сторонний origin (OpenRouter и пр.) не кэшируем вообще.
  - Фолбэк при офлайне без нужной страницы в кэше — отдаём index.html.
- `install`: caches.open + addAll(APP_SHELL) + skipWaiting (активируется
  сразу, не ждёт закрытия вкладок).
- `activate`: удаляет все кэши с префиксом `deliu-cache-v`, кроме текущего,
  затем clients.claim (берёт управление сразу).
- `index.html`: в конце body добавлен блок регистрации SW внутри
  `if ('serviceWorker' in navigator)`, register('service-worker.js') на
  событие window load, лог scope/ошибки в консоль. Остальная разметка
  не тронута.
- Проверки на десктопном Chrome: Console — «SW: зарегистрирован» без ошибок;
  Application → Service Workers — activated and is running; Cache Storage —
  deliu-cache-v1 со всеми файлами; Network → Offline + F5 — страница
  открывается из кэша, История (localStorage) доступна.
- Проверка на проде (Android Chrome, телефон): после загрузки с ?v=18 и
  повторного открытия — режим «В самолёте» → приложение открывается офлайн,
  История на месте и открывается. История офлайн работает, т.к. она в
  localStorage (сеть не нужна) — подтверждено.
- Офлайн НЕ умеет генерировать новый пересказ (нужен OpenRouter) — это

### Этап 15. Часть 5: тема оформления (светлая / тёмная)
- Добавлен переключатель тем. Кнопка `#theme-btn` — ПЕРВАЯ в `.header__actions`

- Логика «рубильника» в `js/app.js` (блок в конце `DOMContentLoaded`):
  атрибут `data-theme` (`light`/`dark`) ставится на `document.documentElement`

  localStorage под ключом `deliu.theme` — пишем НАПРЯМУЮ, не через storage.js
  (сознательное исключение, чтобы не трогать рабочий модуль). Функции:
  `getSavedTheme()` (дефолт — светлая), `applyTheme(theme)` (атрибут + иконка/
  title кнопки + `meta[name=theme-color]`), клик-обработчик (переключает и
  сохраняет), `applyTheme(getSavedTheme())` при загрузке.
- `meta theme-color` меняется вместе с темой: светлая `#1a2a4a`, тёмная
  `#11151c` (константа `THEME_BAR_COLOR`).

- ВАЖНЫЙ УРОК по CSS-селектору (поймали раздвоение фона). Атрибут `data-theme`
  висит на самом `<html>`. Сначала фон корня был задан как `[data-theme="dark"] html`
  — это значит «элемент `html` ВНУТРИ элемента с атрибутом», а корень ни во что
  не вложен, поэтому правило НИКОГДА не срабатывало. Фон красился только на
  `body` (он короче области прокрутки) → при длинном пересказе снизу светлая
  полоса. Правильно целить сам корень через `html[data-theme="dark"]` (без
  пробела = «html, у которого есть этот атрибут»). Итоговые правила:
  `html[data-theme="dark"] { background:#11151c; color:#e6e6e6 }` и
  `html[data-theme="dark"] body { background:#11151c; color:#e6e6e6 }`.
  Остальные правила вида `[data-theme="dark"] .класс` корректны — класс реально
  вложен в html.
